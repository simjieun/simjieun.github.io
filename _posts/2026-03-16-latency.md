## 들어가며

"캐싱을 써야 한다", "CDN은 필수다", "API 직렬 호출은 나쁘다" — 이런 말은 개발하면서 자주 들어왔다.
그런데 왜 그런지 숫자로 설명하라고 하면 한 번도 제대로 답한 적이 없었다. 알고 있다고 생각했지만 사실 아무것도 모르고 있었다.

오늘 처음으로 그 근거가 되는 수치들을 마주했다.
단순히 암기하는 게 아니라, **수치가 설계 결정을 만드는 과정**이 핵심이다.

---

## 레이턴시(Latency)란

레이턴시(Latency)는 **요청을 보낸 순간부터 응답이 돌아올 때까지 걸리는 시간**이다. 한국어로는 "응답지연" 또는 "지연시간"이라고 한다.

스루풋(Throughput)과 혼동하기 쉬운데, 둘은 다른 개념이다.

|          | 의미                               |
| -------- | ---------------------------------- |
| 레이턴시 | 요청 1건이 처리되는 데 걸리는 시간 |
| 스루풋   | 단위 시간당 처리할 수 있는 요청 수 |

레이턴시가 낮다고 스루풋이 높은 것은 아니다. 이 글에서 다루는 수치는 모두 레이턴시다.

## 기본 수치 — 절대값보다 비율로 기억하기

Jeff Dean은 Google의 Senior Fellow로, BigTable, MapReduce, TensorFlow 등 Google 핵심 인프라를 설계한 엔지니어다. 그가 Stanford 강연에서 소개한 아래 수치들은 시스템 설계의 기준값으로 업계 전반에 널리 쓰이고 있다.

| 연산                             | 시간    |
| -------------------------------- | ------- |
| L1 캐시 참조                     | 0.5 ns  |
| L2 캐시 참조                     | 7 ns    |
| 메인 메모리 참조                 | 100 ns  |
| 메모리에서 1MB 순차 읽기         | 0.25 ms |
| 같은 데이터센터 왕복 (RTT)       | 0.5 ms  |
| 네트워크에서 1MB 순차 읽기       | 10 ms   |
| 디스크 탐색 (Seek)               | 10 ms   |
| 디스크에서 1MB 순차 읽기         | 30 ms   |
| 대륙간 패킷 왕복 (CA ↔ 네덜란드) | 150 ms  |

절대값을 외우는 것보다 **비율**로 기억하는 게 훨씬 실용적이다.

1MB 순차 읽기 기준으로 메모리(0.25ms)를 1로 놓고 나누면 비율이 나온다.

```
메모리   0.25ms ÷ 0.25 = 1
네트워크   10ms ÷ 0.25 = 40
디스크     30ms ÷ 0.25 = 120  ← 순차 읽기만 (seek 제외)

→ 메모리 : 네트워크 : 디스크(순차) = 1 : 40 : 120
```

단, 디스크는 seek(10ms)이 항상 추가로 붙는다.
실제 파일 하나를 읽을 때는 seek 10ms + 읽기 30ms = **40ms**, 메모리 대비 **160배**다.
위 비율은 순차 읽기 속도만 비교한 것이고, 실전에서는 seek 비용까지 포함해야 한다.

### 직관을 위한 변환 — 1ns를 1초로 환산하면

숫자가 추상적으로 느껴질 때 이 환산이 도움이 된다.

| 연산        | 실제 시간 | 1ns = 1초 기준 |
| ----------- | --------- | -------------- |
| L1 캐시     | 0.5 ns    | 0.5초          |
| 메인 메모리 | 100 ns    | 1분 40초       |
| 디스크 seek | 10 ms     | **4개월**      |
| 대륙간 RTT  | 150 ms    | **4.75년**     |

메모리와 디스크의 차이가 "1분 40초 vs 4개월"로 느껴지면 캐싱이 왜 필수인지 직관적으로 와닿는다.

## back-of-the-envelope 계산이란

영어 그대로 번역하면 "봉투 뒷면 계산"이다. 미국에서 봉투 뒷면 같은 작은 종이에 즉석으로 계산하던 관행에서 온 말로, 정밀한 도구 없이 빠르게 어림잡는 방식을 뜻한다.

시스템 설계에서는 **정확한 구현 없이 기준 수치 × 규모로 빠르게 추정**하는 기법이다. Jeff Dean이 강조한 것처럼, 엔지니어가 갖춰야 할 핵심 능력 중 하나는 실제로 만들어보기 전에 계산만으로 설계 대안을 비교하는 것이다.

계산 결과가 10배 이상 차이 나면 굳이 구현해보지 않아도 방향이 결정된다. 30배 차이가 나면 논쟁할 필요도 없다.

### 예: 297kB 이미지를 읽는 데 얼마나 걸릴까?

```
297kB = 0.297MB

네트워크: 10ms/MB × 0.297 ≈ 2.97ms
메모리:   0.25ms/MB × 0.297 ≈ 0.074ms
디스크:   30ms/MB × 0.297 + 10ms(seek) ≈ 18.9ms
```

메모리 캐시 vs 디스크가 **255배** 차이다.
이 숫자 하나가 "캐싱을 써야 한다"는 결정의 근거가 된다.

### 예: 썸네일 30개를 보여주는 페이지

```
직렬 로딩:  30 × 10ms(seek) + 30 × 256KB/30MB/s ≈ 560ms
병렬 로딩:  10ms(seek) + 256KB/30MB/s             ≈ 18ms
```

30배 차이. 직렬 방식은 계산만으로 탈락이다.

## 프론트엔드 실무에 직접 연결되는 포인트

### 1. 이미지 로딩 — 병렬 + CDN이 기본 구조

이미지가 많은 페이지에서 로딩 전략에 따라 수치가 크게 달라진다.

```
직렬 로딩 (20장):  20 × ~7ms = 140ms 이상 (TCP/TLS 포함 시 수백 ms)
HTTP/2 병렬:       가장 느린 1개 기준 ~20ms
CDN 캐시 히트:     엣지 RTT ~10ms로 끝
```

원본 이미지를 WebP로 리사이즈해서 서빙하는 이유도 수치로 설명된다.
전송 크기를 줄이면 `10ms/MB × 크기`가 직접 줄어들기 때문이다.

### 2. API 호출 — 직렬 체이닝의 비용

```javascript
// ❌ 나쁜 예: 직렬 await 체이닝 → RTT × 3
const user = await fetchUser();
const perms = await fetchPerms(user.id);
const data = await fetchData(perms);
```

```javascript
// ✅ 좋은 예: 병렬 처리 → 가장 느린 API 1개 기준
const [user, perms, data] = await Promise.all([
  fetchUser(),
  fetchPerms(),
  fetchData(),
]);
```

의존성 없는 요청은 무조건 병렬이다.
React Query의 `staleTime` 설정으로 캐시 히트 시 네트워크 RTT 자체를 0으로 만드는 것이 레이턴시 수치상 가장 극적인 개선이다.

### 3. SSR이 FCP에 유리한 이유 — 라운드트립 횟수

```
CSR: HTML 수신 → JS 번들 다운로드 → API 호출 → 렌더링
     = RTT × 3 + 번들 실행 = 500ms ~ 2초+

SSR: 요청 → 서버 내부 DB 조회(DC RTT 0.5ms) + 렌더링 → 완성된 HTML 수신
     = RTT × 1 + 서버 처리 = 100~400ms
```

서버-서버 간 RTT는 0.5ms로 극히 작다.
클라이언트-서버 왕복이 1번 vs 3번의 차이가 FCP를 결정한다.

### 4. CDN — 코드로 줄일 수 없는 레이턴시

```
한국 유저 → 미국 origin:    ≈ 150ms RTT (빛의 속도 한계, 최적화 불가)
한국 유저 → 한국 CDN 엣지:  ≈ 5~20ms RTT
```

대륙간 RTT는 물리 법칙의 영역이다.
정적 자산을 CDN에 올리는 건 "성능 개선" 옵션이 아니라 구조적 필수 조건이다.

## 앞으로 꼭 염두에 두어야 할 것

### 첫째, 데이터의 위치가 레이턴시를 결정한다

코드의 로직보다 "데이터가 어디에 있는가"가 성능을 먼저 결정한다. 메모리에 있으면 0.25ms, 디스크에 있으면 40ms, 다른 서버에 있으면 네트워크 RTT가 추가된다. 같은 데이터라도 어디서 읽느냐에 따라 수십~수백 배 차이가 난다.

이 관점에서 캐싱은 단순한 최적화 기법이 아니라, 데이터의 위치를 바꾸는 구조적 결정이다. "캐싱을 쓸까 말까"가 아니라 "이 데이터를 어디에 두는 게 맞는가"라는 질문으로 바꿔야 한다.

### 둘째, 라운드트립 횟수가 레이턴시를 지배한다

전송 속도를 높이는 것보다 왕복 횟수 자체를 줄이는 것이 훨씬 효과적이다. 클라이언트-서버 RTT가 50ms일 때, API를 3번 직렬로 호출하면 RTT만 150ms가 된다. 병렬로 바꾸면 코드 한 줄 변경으로 RTT 2번을 제거할 수 있다.

SSR이 FCP에 유리한 이유, React Query 캐싱이 극적인 개선을 만드는 이유, BFF 패턴이 존재하는 이유 — 모두 라운드트립 횟수를 줄이기 위함이다. 어떤 최적화를 선택하든 "이 결정이 라운드트립을 몇 번 줄이는가"를 먼저 계산하는 습관이 필요하다.

### 셋째, 코드로 해결할 수 없는 레이턴시가 있다

대륙간 RTT 150ms는 빛의 속도 한계다. 아무리 코드를 최적화해도 물리적 거리를 줄일 수 없다. 이런 한계를 모르면 코드 레벨에서 엉뚱한 곳에 시간을 낭비한다.

한계를 알아야 올바른 해결책이 보인다. 어떤 문제가 코드로 풀 수 있는 것인지, 인프라 구조의 문제인지를 수치로 먼저 판단할 수 있어야 한다.

## 마무리

오늘 가장 인상 깊었던 수치는 **메모리와 디스크의 160배 차이**였다. 같은 기계 안에 있는 데이터인데, 어디에 저장되어 있느냐만으로 0.25ms와 40ms가 갈린다. 캐싱이 왜 중요한지 수백 번 들었지만, 이 숫자를 보고 나서야 비로소 실감이 됐다.

두 번째로 인상 깊었던 건 **대륙간 RTT 150ms가 빛의 속도 한계**라는 사실이다. 코드를 아무리 잘 짜도 물리적 거리는 극복할 수 없다. 그동안 CDN을 "성능 개선 옵션" 정도로 생각했는데, 이 수치를 보고 나면 글로벌 서비스에서 CDN은 선택이 아니라 구조적 전제 조건이라는 걸 알 수 있다.

이 수치들은 실무에서 이렇게 활용할 수 있다. 이미지가 많은 페이지를 개발할 때 "병렬 로딩을 써야 한다"는 주장 대신 "직렬로 20개를 로드하면 최소 140ms, 병렬이면 20ms"라고 계산해서 말할 수 있다. API 설계 리뷰에서 직렬 체이닝이 나오면 "RTT가 N번 추가됩니다"라는 구체적인 근거로 대안을 제시할 수 있다. 성능 이슈를 디버깅할 때도 어디가 병목인지 수치로 가설을 세우고 접근할 수 있다.

무엇보다 오늘 얻은 것은 **논리적인 근거를 갖추게 됐다**는 점이다. "좋다고 하니까", "보통 이렇게 하니까"가 아니라, 숫자에서 출발해서 결론을 도출하는 과정을 이해했다. 프론트엔드 개발을 하면서 인프라나 네트워크를 직접 다루지 않더라도, 이 수치들을 알고 있으면 왜 이 구조가 빠른지 설명할 수 있는 언어가 생긴다. 그게 오늘의 가장 큰 수확이다.
